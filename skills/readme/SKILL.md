# README 编写与修订

## 核心原则

从用户视角编写，回答三个问题：**这是什么？怎么用？有什么特点？**

**关键要求**：
- ✅ 内容与源码一致，不虚构配置或功能
- ✅ 子模块/包级 README 不重复根 README 的共享内容
- ✅ 示例可直接运行，配置项真实存在
- ✅ 只说明项目功能，第三方框架能力不展开
- ✅ 语言简洁、术语统一（如"模块"vs"子项目"需统一）

## 执行流程

### 步骤1：确定范围和层级

询问用户要处理哪个 README（若未明确）：
- 根目录：`README.md`
- 子模块：`module-name/README.md`  
- 包级：`src/.../package-name/README.md`

### 步骤2：读取上级 README（子模块/包级）

#### 2.1 子模块/包级：读取上级

| 当前层级 | 必须读取 |
|---------|---------|
| 子模块 | 根目录 README |
| 包级 | 所属子模块 README（无则读根目录） |

#### 2.2 根目录：跳过

此步骤不适用。

### 步骤3：核实能力并起草

#### 3.1 阅读源码/配置

确认实际功能、配置项、API。

#### 3.2 按层级结构起草

参照下方「分层结构」要求：

1. 先说「这是什么」（标题、描述、功能）
2. 再说「怎么用」（安装、快速开始、代码示例）
3. 最后补充配置、注意事项

**正常起草，不用担心与上级重复**（去重在步骤4进行）。

### 步骤4：对比去重（子模块/包级）

**如果是子模块或包级 README**，执行以下去重步骤：

#### 4.1 核心原则

将起草的内容与步骤2读取的上级 README 逐段对比，只要内容在上级已说明，当前 README 就删除，不保留任何重复。

#### 4.2 常见重复类型

包括但不限于：环境要求、版本兼容性、技术栈、配置约定、构建命令、架构说明、许可证等。

完整清单见 [reference.md](./reference.md#共享内容清单)。

#### 4.3 添加引用

删除重复内容后，必要时添加相对路径引用上级 README。

---

**如果是根目录 README**，跳过此步骤。

### 步骤5：自检质量

#### 5.1 核心质量（MUST）

- [ ] 用户能在 30 秒内理解项目用途
- [ ] 核心功能有可运行示例
- [ ] 配置说明与源码一致
- [ ] 子模块/包级未重复上级 README 的内容（已在步骤4完成）

#### 5.2 可选质量（SHOULD/MAY）

- [ ] 根目录注明版本与兼容性
- [ ] 较长 README（>200行）有目录
- [ ] 常见问题有解决方案

### 步骤6：语言精简

在保持原意前提下优化表达（简洁、主动语态、术语统一）。详见 [reference.md](./reference.md#语言与风格)。

### 步骤7：交付

回复用户时简要说明已确认的检查项。

## 分层结构

按目标读者调整详略。详细模板和章节说明见 [reference.md](./reference.md)。

| 层级 | 目标读者 | 必含章节 | 不应包含 |
|------|---------|---------|---------|
| **根目录** | 项目使用者 | 标题描述 · 功能特点 · 环境要求 · 安装依赖 · 快速开始 · 子项目概览 · 技术栈 · 版本兼容性 | - |
| **子模块** | 库使用者 | 标题描述 · 功能模块 · 快速使用（依赖+示例） · 配置说明 | 环境要求 · 技术栈 · 版本兼容性（引用根README） |
| **包级** | 开发者 | 标题 · 职责 · 用法要点/入口类 | 环境 · 版本 · 技术栈（引用上级） |

## 适用范围

根目录、子模块、package 级（如 `src/.../某包/README.md`）的 README 均适用。

## 更新时机

建议在以下情况更新 README：
- 新增/修改核心功能或 API
- 依赖版本、环境要求、兼容性变化
- 配置项增删改
- 用户反馈"文档与实际不符"时
